3) Step-by-step (Real Commands)

This is the exact workflow the script automates. I keep these steps documented so the process is transparent and reviewable. 

# A) Identify system & baseline
uname -a
( command -v lsb_release >/dev/null 2>&1 && lsb_release -a ) || cat /etc/os-release
uptime

# B) CPU / Memory / Disk
top -b -n1 | head -n 30
free -m
df -h
dmesg -T | tail -n 50

# C) Networking
ip a
ip r
ss -tulpn | head -n 80
( command -v resolvectl >/dev/null 2>&1 && resolvectl status ) || cat /etc/resolv.conf

# D) Logs (time boxed)
journalctl -xe --no-pager | tail -n 200
# Service-specific:
journalctl -u nginx --since "2 hours ago" --no-pager | tail -n 300

# E) Security signals
# SSH failures (varies by distro; journalctl is most reliable)
journalctl _SYSTEMD_UNIT=ssh.service --since "24 hours ago" --no-pager | grep -Ei "failed|invalid|authentication|bad protocol" | tail -n 80
# Recent sudo usage
journalctl -t sudo --since "24 hours ago" --no-pager | tail -n 80

4) Script Usage

# Basic: last 2 hours
bash triage.sh --since "2h"

# Service mode: focus on a unit + config checksum (example: nginx)
bash triage.sh --since "2h" --service nginx

# Create a bundle
bash triage.sh --since "6h" --service nginx --bundle

6) “Bad Smell” Detection (Severity Tags)

The script flags common issues early. The goal is not “AI,” it’s fast triage: detect obvious root cause candidates immediately.

Disk pressure

HIGH if any mount > 85%.

OOM / memory

HIGH if dmesg shows OOM kills or free memory is critically low.

Restart loops

MED/HIGH if systemd shows repeated restarts within the timeframe.

7) Validation Steps (How I Prove It Works)

I validate this tool like production automation: predictable outputs, safe permissions, no secrets leaking, and repeatable tests with known scenarios. 

# 1) Dry-run safety: ensure script is read-only
# - no rm/mv/chmod on system paths
# - only writes under triage_* output dir

# 2) Run on a known-good host (baseline)
bash triage.sh --since "1h"
# Expect: no HIGH severity flags, report still generated

# 3) Create a controlled failure (non-destructive)
# Example A: fill a temp filesystem (if safe in your lab)
# Example B: restart a test service repeatedly and confirm "restart loop" flag
for i in $(seq 1 5); do sudo systemctl restart cron; sleep 2; done
bash triage.sh --since "15m" --service cron

# 4) Validate JSON format
# (use jq if available)
command -v jq >/dev/null 2>&1 && jq . triage_*/report.json >/dev/null

# 5) Secret hygiene check (basic)
# Ensure report files do not contain tokens/passwords
grep -RniE "token|password|secret|authorization: bearer" triage_* || true
