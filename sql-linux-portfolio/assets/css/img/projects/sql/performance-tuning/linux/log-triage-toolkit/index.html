Linux Case Study: Log Triage Toolkit

A repeatable incident workflow for quickly narrowing down the root cause using journald, service status, and targeted filtering. Replace sample outputs with real screenshots and logs as you build.

Download commands.md Download triage.sh Back to Linux

1) Goal

Provide a consistent playbook for investigating service failures (timeouts, restarts, 5xx errors), capturing evidence, and producing a short RCA summary.

2) Environment

OS: Ubuntu / RHEL (document your version)
Logging: journald + /var/log/*
Tools: journalctl, systemctl, ss, ip, df, free, top, grep, awk

3) Step-by-step Workflow

Step A: Confirm service state

Identify the failing unit and the last restart reason.

Step B: Pull focused logs

Filter by unit and timeframe to reduce noise.

Step C: Validate system health

Check CPU/mem/disk, network sockets, DNS, and dependencies.

# Step A: Service health
systemctl status myservice --no-pager
systemctl is-failed myservice
systemctl show -p ActiveEnterTimestamp myservice

# Step B: Logs (last 2 hours)
journalctl -u myservice --since "2 hours ago" --no-pager
journalctl -u myservice -p warning --since "2 hours ago" --no-pager

# Step C: Quick system checks
df -h
free -m
uptime
ss -tulpn | head
ip a
resolvectl status 2>/dev/null || cat /etc/resolv.conf

4) Scripted Triage (optional)

Include a script to standardize evidence capture. Keep it safe by default (read-only) and clearly label outputs.

5) What Broke + Fix

Add a real example here. Format: 

Symptom:
- service restarting every 30s
- upstream returning 502

Root Cause:
- config typo in /etc/myservice/config.yml
- missing env var after deploy

Fix:
- corrected config
- systemctl daemon-reload
- restarted service
- validated logs clean for 30 minutes

Prevention:
- add config validation in CI
- add health check + alert thresholds
