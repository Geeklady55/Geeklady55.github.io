SQL Case Study: True Top Customers by Net Revenue Retention (NRR)

This case study demonstrates a real-world, SaaS-style retention metric: month-over-month revenue retention per customer, with churn/expansion/contraction classification and ranking. I wrote this the same way I would for production support: correct math, edge-case handling, and validation.

1) Use Case

The business question isn’t “top customers by revenue.” It’s “top customers by revenue retention.” This catches: churn (lost revenue), expansion (upsells), contraction (downgrades), and new starts. The output is interview-grade but also production-friendly: it’s traceable, explainable, and testable.

2) Realistic Schema

This version uses a minimal invoice fact table, but it maps cleanly to SaaS billing systems (Stripe-like invoices, ERP invoice lines, etc.). If you have invoice lines, you’d aggregate line totals into invoices first. 

-- PostgreSQL-style DDL (adaptable)
CREATE TABLE invoices (
  invoice_id      bigserial PRIMARY KEY,
  customer_id     bigint NOT NULL,
  invoice_date    date   NOT NULL,
  amount          numeric(12,2) NOT NULL,
  currency        text DEFAULT 'USD',
  status          text DEFAULT 'paid'
);

-- Helpful indexes for scale:
CREATE INDEX IF NOT EXISTS idx_invoices_customer_date ON invoices(customer_id, invoice_date);
CREATE INDEX IF NOT EXISTS idx_invoices_date ON invoices(invoice_date);

4) Final Query

Key details: correct temporal aggregation, LAG for prev month, ratio math with NULL/0 protection, movement classification, and ranking that prefers retention first, then revenue impact. 

WITH monthly_rev AS (
  SELECT
    customer_id,
    DATE_TRUNC('month', invoice_date) AS month,
    SUM(amount) AS mrr
  FROM invoices
  -- WHERE status = 'paid'  -- enable if you store non-paid invoices
  GROUP BY 1, 2
),
with_prev AS (
  SELECT
    customer_id,
    month,
    mrr,
    LAG(mrr) OVER (PARTITION BY customer_id ORDER BY month) AS prev_mrr
  FROM monthly_rev
),
nrr AS (
  SELECT
    customer_id,
    month,
    prev_mrr,
    mrr,
    CASE
      WHEN prev_mrr IS NULL OR prev_mrr = 0 THEN NULL
      ELSE ROUND((mrr / prev_mrr)::numeric, 4)
    END AS nrr_ratio,
    (mrr - COALESCE(prev_mrr, 0)) AS delta_mrr,
    CASE
      WHEN prev_mrr > 0 AND mrr = 0 THEN 'churn'
      WHEN prev_mrr = 0 AND mrr > 0 THEN 'new'
      WHEN mrr > prev_mrr THEN 'expansion'
      WHEN mrr < prev_mrr THEN 'contraction'
      ELSE 'flat'
    END AS movement_type
  FROM with_prev
)
SELECT
  customer_id,
  month,
  prev_mrr,
  mrr,
  nrr_ratio,
  delta_mrr,
  movement_type,
  DENSE_RANK() OVER (
    ORDER BY nrr_ratio DESC NULLS LAST, delta_mrr DESC
  ) AS nrr_rank
FROM nrr
WHERE prev_mrr IS NOT NULL
ORDER BY nrr_rank, month DESC;

5) Production Notes

Data correctness

NRR is computed only when prev_mrr is non-zero; new-start months get NULL ratio by design.

Time zones

In production, normalize invoice_date to UTC-based business day rules before monthly bucketing if needed.

Scale

Indexes on (customer_id, invoice_date) keep monthly aggregation fast; consider partitioning by date for very large tables.

6) Validation Steps

This is how I validate before trusting retention results: compare monthly totals, compare customer-month row counts, spot-check edge cases (new, churn, expansion), and validate that month bucketing does not drop invoices. 

-- A) Monthly totals should match source totals
WITH src AS (
  SELECT DATE_TRUNC('month', invoice_date) AS month, SUM(amount) AS src_total
  FROM invoices
  GROUP BY 1
),
agg AS (
  SELECT month, SUM(mrr) AS agg_total
  FROM (
    SELECT customer_id, DATE_TRUNC('month', invoice_date) AS month, SUM(amount) AS mrr
    FROM invoices
    GROUP BY 1,2
  ) x
  GROUP BY 1
)
SELECT s.month, s.src_total, a.agg_total, (s.src_total - a.agg_total) AS diff
FROM src s
JOIN agg a USING (month)
ORDER BY s.month;
